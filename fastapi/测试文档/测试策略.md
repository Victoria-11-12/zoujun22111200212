# 测试策略

## 一、概述

### 1.1 项目背景

本项目是一个基于 FastAPI + LangChain + LangGraph 的智能电影数据分析系统，核心能力包括：

- 用户自然语言查询电影数据（SQL Agent 自动生成并执行 SQL）
- 管理员批量操作与回滚（Admin Agent 工具调用链）
- 在线图表生成（LangGraph 工作流：SQL查询 → 代码生成 → 安全评估 → Docker沙箱执行）
- AI 回复质量评估（异步批量评估 + 多维度评分）

### 1.2 测试目标

| 目标 | 指标 |
|------|------|
| 代码覆盖率 | ≥ 80%（当前 85%） |
| 测试通过率 | 100%（核心路径必须全过） |
| 测试分层 | 单元测试 / 集成测试 / 端到端测试 三层覆盖 |
| 报告产出 | HTML 测试报告 + 覆盖率报告 + 测试日志 |

### 1.3 测试范围

- **包含**：`app/` 目录下所有 Python 模块（models、tools、chains、agents、workflows、services、routers、config、history、logs）
- **不包含**：前端代码（`Web_Node/`）、Dockerfile、配置文件模板

---

## 二、测试分层策略

采用经典测试金字塔模型，自底向上分为三层：

```
        ┌─────────┐
        │  E2E    │  20 个场景，覆盖 4 个 API 路由
        │  端到端  │  耗时约 3 分钟
        ├─────────┤
        │ 集成测试 │  35 个用例，覆盖 chains/agents/workflows/services
        │         │  耗时约 5 分钟
        ├─────────┤
        │ 单元测试 │  45 个用例，覆盖 models/tools/workflows纯函数/constants
        │         │  耗时约 5 秒
        └─────────┘
```

### 2.1 单元测试（Unit Tests）

**定位**：验证单个函数/类的逻辑正确性，不依赖外部服务。

**覆盖范围**：

| 模块 | 测试内容 | Mock 策略 |
|------|---------|-----------|
| `app/models.py` | Pydantic 模型字段验证、默认值、必填校验、类型检查 | 无需 Mock |
| `app/history.py` | 会话历史增删查、截断逻辑、LangChain 消息封装 | 无需 Mock |
| `app/tools/admin_tools.py` | 危险关键词检测、SQL 安全检查、工具列表完整性 | 纯函数无需 Mock；数据库操作需 Mock |
| `app/tools/sql_tools.py` | SQL 查询执行、参数传递 | Mock 数据库 |
| `app/tools/web_tools.py` | 子进程调用、超时处理、输出解析 | Mock subprocess |
| `app/workflows/chart_workflow.py` | 代码提取正则、静态安全评估、条件路由逻辑 | 无需 Mock |
| `app/logs.py` | SQL 构造、参数传递、异常处理 | Mock 数据库连接 |
| `app/config.py` | LLM/数据库/连接池初始化参数 | Mock 环境变量 |
| `app/agents/sql_agent.py` | 工具列表完整性 | 无需 Mock |
| `app/services/analyst_service.py` | 进度状态初始值、状态转换、线程锁 | 无需 Mock |

**断言原则**：

- 纯函数：精确断言返回值
- 数据库操作：Mock 后断言调用参数和次数
- 模型验证：断言 Pydantic ValidationError 的触发条件和字段路径

### 2.2 集成测试（Integration Tests）

**定位**：验证多个组件协作的正确性，需要真实的外部依赖（LLM、数据库、Docker）。

**覆盖范围**：

| 层级 | 测试对象 | 依赖 |
|------|---------|------|
| Chains 层 | 意图分类链、回复生成链、SQL 结果包装链、警告链、评估链 | 真实 LLM |
| Agents 层 | SQL Agent 工具调用链、Admin Agent 批量操作与回滚 | 真实 LLM + 数据库 |
| Workflows 层 | LangGraph 图拓扑、节点注册、边连接、条件路由、完整编排 | 真实 LLM + Docker |
| Services 层 | 用户/管理员/图表/评估服务的业务编排、异步并发评估 | 真实 LLM + 数据库 + Docker |

**断言原则**：

由于 LLM 输出具有不确定性，集成测试采用宽松断言策略：

| 断言类型 | 适用场景 | 示例 |
|---------|---------|------|
| 关键字匹配 | 意图分类链 | `"NEED_SQL" in result` |
| 非空检查 | 生成类链 | `len(result.strip()) > 0` |
| 结构化字段检查 | 评估链（Pydantic 输出） | `1 <= result.score <= 5` |
| 状态字段检查 | 工作流节点 | `"sql_result" in state` |
| 路由结果检查 | 条件路由 | `result == "pythonagent"` |
| SSE 格式检查 | 流式服务 | `"[DONE]" in chunks` |

### 2.3 端到端测试（E2E Tests）

**定位**：验证从 HTTP 请求到响应的完整链路，使用真实的所有外部依赖。

**覆盖范围**：

| 接口 | 场景数 | 覆盖路径 |
|------|--------|---------|
| `POST /api/ai/stream` | 6 | 正常对话、SQL查询、数据库无结果→浏览器搜索、安全威胁、多轮对话、异常处理 |
| `POST /api/admin/ai/stream` | 5 | 正常管理、批量操作、回滚、安全威胁、危险SQL拦截 |
| `POST /api/chart/generate` | 5 | 非图表请求、图表生成成功、代码生成失败重试、沙箱执行失败、危险代码拦截 |
| `POST/GET /api/analyst/*` | 4 | 启动评估、进度查询、结果查询、统计查询 |

**断言原则**：

- HTTP 状态码必须为 200
- Content-Type 必须为 `text/event-stream`
- 流式响应必须以 `[DONE]` 标记结束

## 三、测试环境

### 3.1 环境配置

| 配置项 | 值 | 说明 |
|--------|-----|------|
| Python 版本 | 3.10 | 与生产环境一致 |
| 测试框架 | pytest 8.3.5 | 异步支持（asyncio_mode=auto） |
| 覆盖率工具 | coverage.py 7.13.5（pytest-cov） | HTML + 终端双输出 |
| 报告插件 | pytest-html 4.2.0 | 自包含 HTML 报告 |
| 环境变量 | `.env.test` | 测试专用，通过 `load_dotenv(override=True)` 覆盖 |
| 数据库 | MySQL（movie_test 库） | 端口 3307，独立于生产库 |
| Docker | pyecharts-sandbox 镜像 | 图表代码沙箱执行，256MB 内存限制，网络隔离 |

### 3.2 测试数据管理

- 使用独立的测试数据库 `movie_test`，与生产数据库隔离
- 测试前不自动清理数据（E2E 测试依赖已有数据）
- 管理员批量操作测试后通过回滚恢复数据
- 评估测试写入 `eval_results` 表，可手动清理

## 四、执行策略

### 4.1 执行顺序

```
第一阶段：单元测试（5 秒）
  → models → history → tools安全检测 → workflows纯函数 → constants → tools → services

第二阶段：集成测试（5 分钟）
  → chains → agents → workflows → services

第三阶段：端到端测试（3 分钟）
  → user → admin → chart → analyst
```

### 4.2 运行命令

```bash
# 运行全部测试（生成报告 + 覆盖率）
pytest

# 按层级运行
pytest -m unit
pytest -m integration
pytest -m e2e

# 按模块运行
pytest tests/unit/models/
pytest tests/integration/chains/
pytest tests/e2e/test_user_router.py
```

### 4.3 报告产出

| 产物 | 路径 | 说明 |
|------|------|------|
| 测试报告 | `test_results/reports/report_YYYYMMDD_HHMMSS.html` | 带时间戳的自包含 HTML |
| 覆盖率报告 | `test_results/coverage/index.html` | 文件级/函数级/类级覆盖率 |
| 测试日志 | `test_results/logs/test.log` | DEBUG 级别，含时间戳 |

## 五、质量门禁

### 5.1 提交前检查

| 检查项 | 标准 | 阻断级别 |
|--------|------|---------|
| 单元测试通过率 | 100% | 阻断 |
| 集成测试通过率 | ≥ 95%（允许 LLM 偶发不稳定） | 警告 |
| E2E 测试通过率 | 100% | 阻断 |
| 代码覆盖率 | ≥ 80% | 警告 |
| 新增代码覆盖率 | ≥ 90% | 阻断 |

### 5.2 覆盖率豁免

以下模块因技术原因允许较低覆盖率：

| 模块 | 当前覆盖率 | 豁免原因 |
|------|-----------|---------|
| `web_tools.py` | 8% | 调用外部命令行工具，单元测试需 Mock subprocess，实际风险低 |
| `analyst_service.py` | 48% | 大量数据库查询和异步编排逻辑，集成测试已覆盖核心路径 |

## 六、已知风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| LLM 输出不稳定 | 集成测试偶发失败（JSON 解析异常） | 重跑即可恢复；评估链使用 `with_structured_output` 兜底 |
| Docker 沙箱不可用 | 图表工作流测试全部失败 | 测试前确认 Docker Desktop 已启动 |
| 数据库连接池耗尽 | 异步并发评估测试超时 | Semaphore(5) 限制并发数 |
| 测试数据库被污染 | E2E 测试结果不可信 | 管理员回滚测试自动恢复；必要时手动清理 |

---

## 七、测试用例设计原则

1. **一个测试只验证一个行为**：避免单个用例覆盖多个不相关的断言
2. **测试用例命名自描述**：函数名直接说明测试场景，如 `test_web_search_fallback`
3. **正常路径优先**：先覆盖核心业务路径，再覆盖边界和异常
4. **LLM 输出不精确断言**：不断言 LLM 的具体回复内容，只断言格式和关键标记
5. **E2E 保持精简**：只覆盖核心用户场景，细化的边界条件放在集成测试层
