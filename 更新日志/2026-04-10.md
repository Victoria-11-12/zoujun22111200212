# 2026-04-10

## AI 模块迁移：从 Flask 到 FastAPI，完善 LangChain 原生异步架构

### 迁移背景

昨天将 AI 模块从 Node.js 迁移到 Flask + LangChain，但 Flask 作为同步框架存在以下限制：
- 不支持原生 `async def`，`astream()` 无法真正并发
- 异步生成器需要 `sync_generator` 包装，代码冗余
- LangChain 的完整异步能力（`ainvoke`、`astream`）无法发挥

今天将 AI 模块从 Flask 迁移到 FastAPI，形成三服务微服务架构。

### 架构变更

#### 迁移前（双服务）
```
前端 demo.html
    ├── Node.js :3000（用户系统、数据 CRUD）
    └── Flask :5000（机器学习 + AI 模块）
```

#### 迁移后（三服务微服务）
```
前端 demo.html
    ├── Node.js :3000（用户系统、数据 CRUD）
    ├── Flask :5000（机器学习预测）
    └── FastAPI :8000（AI 对话、SQL Agent、管理员 Agent）
```

#### 各服务职责

| 服务 | 端口 | 职责 |
|------|------|------|
| Node.js | 3000 | 用户系统、电影 CRUD、留言、日志 |
| Flask | 5000 | 黑马筛选、票房预测（机器学习模型） |
| FastAPI | 8000 | AI 对话、SQL Agent、管理员 Agent、对话日志 |

### 核心改动

#### 改动一：同步全部改为异步

| 对比 | Flask 版 | FastAPI 版 |
|------|---------|-----------|
| 路由函数 | `def ai_stream()` | `async def ai_stream()` |
| LLM 调用 | `.invoke()` | `.ainvoke()` |
| 流式输出 | `.stream()` + `yield` | `.astream()` + `async for yield` |
| 请求体 | `request.json` | Pydantic `ChatRequest` |
| 响应 | `Response(generate())` | `StreamingResponse(generate())` |
| 启动 | `app.run(port=5000)` | `uvicorn.run(port=8000)` |

**删除了 `sync_generator` 包装器**，FastAPI 原生支持异步生成器。

#### 改动二：对话历史注入 LLM

**问题**：AI 不记得之前的对话内容，无法理解指代关系。

**原因**：`direct_chain` 和 `wrap_chain` 的 prompt 模板中没有 `MessagesPlaceholder`，历史消息没有传给 LLM。

**解决**：

```python
# 改前（无历史）
direct_chain = ChatPromptTemplate.from_messages([
    ("system", REPLY_PROMPT),
    ("human", "{message}")
])

# 改后（注入历史）
direct_chain = ChatPromptTemplate.from_messages([
    ("system", REPLY_PROMPT),
    MessagesPlaceholder(variable_name="history"),  # ← 新增
    ("human", "{message}")
])
```

调用时传入历史：

```python
history = get_history(session_id)
async for chunk in direct_chain.astream({"message": message, "history": history}):
    ...
```

管理员接口同样加上历史记忆，保留最近 10 轮对话。

#### 改动三：ChatOpenAI 参数名兼容

**问题**：`httpx.ConnectError: [Errno 11001] getaddrinfo failed`

**原因**：langchain-openai 新版本参数名从 `openai_api_key`/`openai_api_base` 改为 `api_key`/`base_url`，使用新参数名时被静默忽略，默认连接 `api.openai.com`，国内连不上。

**解决**：使用旧参数名确保兼容：

```python
llm = ChatOpenAI(
    model=os.getenv('MODEL_NAME'),
    openai_api_key=os.getenv('API_KEY'),       # 旧参数名
    openai_api_base=os.getenv('API_BASE'),      # 旧参数名
    temperature=0.1
)
```

#### 改动四：SQL Agent 改用 tool-calling 类型

**问题**：`Could not parse LLM output` — DeepSeek 不遵守 ReAct 的 `Final Answer:` 文本格式。

**解决**：

```python
sql_agent = create_sql_agent(
    llm=llm,
    toolkit=user_toolkit,
    agent_type="tool-calling",          # ← 改用 tool-calling
    handle_parsing_errors=True,
)
```

`tool-calling` 通过 Function Calling API 调用工具，不依赖文本格式解析，DeepSeek 完美支持。

#### 改动五：数据库安全约束

**问题**：用户可以通过 SQL Agent 查询任意表（users、logs 等），存在数据泄露风险。

**解决**：用户端使用 `include_tables` 限制数据库连接：

```python
# 管理员：所有表
db = SQLDatabase.from_uri(DB_URI)

# 用户：只有 movies 表
db_user = SQLDatabase.from_uri(DB_URI, include_tables=['movies'])

user_toolkit = SQLDatabaseToolkit(db=db_user, llm=llm)
```

双重保护：prompt 限制 + 数据库层面限制。即使 prompt 被绕过，数据库也会拒绝访问其他表。

#### 改动六：对话日志记录系统

**设计**：双层隔离，AI 上下文记忆和研究日志完全独立。

| | AI 上下文记忆 | 对话日志（研究用） |
|---|---|---|
| 存储位置 | 内存 dict | MySQL |
| 保留轮数 | 最近 10 轮 | 全部记录 |
| 给谁用 | LLM（多轮对话） | 研究分析 |
| 需要 intent 吗 | 不需要 | 需要 |

**建表**：

```sql
-- 用户对话日志
CREATE TABLE user_chat_logs (
    id INT AUTO_INCREMENT PRIMARY KEY,
    session_id VARCHAR(100),
    user_name VARCHAR(50),
    role VARCHAR(20),
    content TEXT,
    intent VARCHAR(20),
    model_name VARCHAR(50),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 管理员对话日志
CREATE TABLE admin_chat_logs (
    id INT AUTO_INCREMENT PRIMARY KEY,
    session_id VARCHAR(100),
    user_name VARCHAR(50),
    role VARCHAR(20),
    content TEXT,
    model_name VARCHAR(50),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**日志记录位置**：放在流式生成器内部，与 `save_history` 同一位置，确保存储的是纯文本而非 SSE 格式字符串。

```python
async def direct_reply_stream(message, session_id, intent=None, user_name=""):
    history = get_history(session_id)
    log_user_chat(session_id, "user", message, intent=intent, user_name=user_name)  # 记用户
    reply = ''
    async for chunk in direct_chain.astream({"message": message, "history": history}):
        reply += chunk
        yield f"data: {json.dumps({'content': chunk}, ensure_ascii=False)}\n\n"
    yield "data: [DONE]\n\n"
    save_history(session_id, message, reply)
    log_user_chat(session_id, "ai", reply, intent=intent, user_name=user_name)  # 记AI
```

### 改动八：前端Markdown渲染优化

**问题**：AI返回的Markdown格式文本在前端显示为纯文本，可读性差。

**方案**：引入marked.js逐块渲染，30ms防抖平衡打字机效果和渲染性能。

**核心改动**（ai.js）：
```javascript
let renderTimer = null;

function appendAIMessage(text) {
    currentAIContent += text;
    clearTimeout(renderTimer);
    renderTimer = setTimeout(() => {
        messageContent.innerHTML = marked.parse(currentAIContent);
    }, 30);  // 防抖渲染
}
```

**效果**：保留打字机效果，Markdown格式（表格、列表、粗体）实时生效。


### 技术要点梳理

#### ReAct 与 Function Calling 的关系

- **ReAct** = 设计模式（思考 → 行动 → 观察 → 循环）
- **Function Calling** = 技术手段（模型调用工具的方式）
- 两者配合实现：ReAct 用 Function Calling 来完成"行动"这一步
- 普通用户和管理员都是 ReAct Agent，只是工具不同

#### SQL Agent vs Tool Calling Agent

| 对比项 | create_sql_agent（用户） | create_tool_calling_agent（管理员） |
|--------|------------------------|----------------------------------|
| SQL 权限 | 只读 SELECT | SELECT + DELETE + UPDATE |
| 可操作的表 | movies（include_tables） | 所有表 |
| 额外工具 | 无 | create_user（bcrypt 加密） |
| 循环次数决定者 | Agent 自己（信息够了就停） | 用户指令（几个任务就循环几次） |

#### 同步/异步/流式的区别

| | 同步 invoke | 异步 ainvoke | 流式 stream |
|---|---|---|---|
| 等时能干别的吗 | 不能 | 能 | 能 |
| 用户何时看到内容 | 全部生成完 | 全部生成完 | 第一个字就看到 |
| 适合场景 | SQL Agent（无流式） | FastAPI 高并发 | 聊天界面 |

#### RunnableBranch 为什么没用

RunnableBranch 期望每个分支返回相同类型的值，不能返回生成器（SSE 流）。改用 if/else 在路由层做意图判断和链路选择。这是业界标准做法（ChatGPT、Claude 等都这样做）。
