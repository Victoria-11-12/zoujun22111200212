## CodeAct Agent 在线绘图模块实现

### 模块背景

在已有 AI 助手（SQL Agent + 意图判断链）的基础上，新增"在线绘图"模块，实现 CodeAct Agent 架构：AI 根据用户需求自动查询数据库、生成 pyecharts 代码并执行，返回 ECharts 交互式图表。

### 架构设计

```
用户输入 "绘制2012年票房排行前十的折线图"
    │
    ▼
chart_intent_chain（意图判断）
    ├── NOT_CHART → chart_not_chain（直接回复）
    └── IN_CHART → chart_generate_stream
                      │
                      ▼
                  sql_agent（查数据）
                      │
                      ▼
              python_chart_chain（生成代码）
                      │
                      ▼
                  exec() 执行代码
                      │
                      ▼
              render_embed() → iframe 渲染
```

### 核心组件

| 组件 | 类型 | 功能 |
|------|------|------|
| chart_intent_chain | LLM Chain | 判断用户是否需要绘图（IN_CHART / NOT_CHART） |
| chart_not_chain | LLM Chain | 不需要绘图时的友好回复 |
| python_chart_chain | LLM Chain | 根据需求和 SQL 结果生成 pyecharts 代码 |
| sql_agent | AgentExecutor | 复用用户模块的 SQL Agent 查询电影数据 |
| chart_generate_stream | 异步生成器 | 编排整个流程，SSE 流式输出 |

### 遇到的关键问题与解决方案

#### 问题一：重复定义 ChatRequest 导致接口冲突

**现象**：`/api/ai/stream` 接口突然收不到 `message` 字段。

**原因**：在线绘图模块定义了第二个 `class ChatRequest`，字段名从 `message` 改成了 `chart_message`，Pydantic 后定义的类覆盖了先定义的，导致 AI 助手接口解析失败。

**解决**：新建独立的 `ChartRequest` 类，字段名保持 `message` 不变。

```python
# 之前（冲突）
class ChatRequest(BaseModel):
    chart_message: str  # 覆盖了 AI 助手的 ChatRequest

# 之后（独立）
class ChartRequest(BaseModel):
    message: str  # 与前端字段对齐
    sessionId: str = ""
```

#### 问题二：LLM 输出包含 markdown 标记导致 exec() 语法错误

**现象**：`SyntaxError: invalid syntax (<string>, line 1)`，错误指向 ` ```python `。

**原因**：LLM 返回的代码带有 ` ```python ... ``` ` markdown 标记，`exec()` 直接执行原始字符串，Python 无法解析 markdown 标记。

**解决**：在 exec 之前用正则提取代码块。

```python
import re
code_match = re.search(r'```python\s*(.*?)```', code, re.DOTALL)
if code_match:
    code = code_match.group(1).strip()
```

#### 问题三：pyecharts API 版本不兼容（set_global_options）

**现象**：`ImportError: cannot import name 'set_global_options' from 'pyecharts.options'`

**原因**：`set_global_options` 是 pyecharts 1.x 的全局函数，2.x 已删除。prompt 中要求模型使用该 API，模型照做但执行时报错。

**解决**：修改 prompt，明确告知使用 2.x 的 `chart.set_global_opts()` 实例方法，禁止使用 `set_global_options`。

#### 问题四：InitOpts 不支持 font_family 参数

**现象**：`TypeError: InitOpts.__init__() got an unexpected keyword argument 'font_family'`

**原因**：pyecharts 2.x 的 `InitOpts` 没有 `font_family` 参数，prompt 中要求模型设置该参数。

**解决**：从 prompt 中彻底移除字体设置相关要求，浏览器默认即可渲染中文。

#### 问题五：Windows 不支持 /tmp/ 目录

**现象**：`FileNotFoundError: [Errno 2] No such file or directory: '/tmp/chart.html'`

**原因**：prompt 要求模型使用 `render('/tmp/chart.html')` 写文件，但 Windows 没有 `/tmp/` 目录。

**解决**：改用 `render_embed()` 直接返回 HTML 字符串，赋值给变量 `CHART_HTML`，通过 `exec()` 的 `local_vars` 获取，完全不依赖文件系统。

```python
local_vars = {}
exec(code, {"__builtins__": __builtins__}, local_vars)
chart_html = local_vars.get('CHART_HTML', '')
```

#### 问题六：iframe 中 ECharts 未加载（代理拦截 CDN）

**现象**：`echarts.min.js:1 Failed to load resource: net::ERR_PROXY_CONNECTION_FAILED`，`echarts is not defined`。

**原因**：后端将 `render_embed()` 的片段包成完整 HTML 时，引用了 CDN（`cdn.jsdelivr.net`），用户浏览器开了代理，iframe 的 srcdoc 加载 CDN 被拦截。

**解决**：改用本地 ECharts 文件，引用 Node.js 静态服务器上的 `js/echarts.js`。

```python
# 之前（CDN，被代理拦截）
<script src="https://cdn.jsdelivr.net/npm/echarts@5/dist/echarts.min.js">

# 之后（本地文件，不走代理）
<script src="http://localhost:3000/js/echarts.js">
```

#### 问题七：render_embed() 只返回 HTML 片段，缺少 ECharts 初始化

**现象**：iframe 中只有一个可拖动的滚动条，没有图表。

**原因**：`render_embed()` 只返回 `<div>` 片段和内联 `<script>`，不包含 ECharts 库本身。需要包成完整 HTML 页面并引入 ECharts。

**解决**：后端拿到片段后包一层完整 HTML，引入 ECharts 并自动初始化所有图表实例。

```python
chart_html = f"""<!DOCTYPE html>
<html><head><meta charset="utf-8">
<script src="http://localhost:3000/js/echarts.js"><\/script>
<style>html,body{{margin:0;padding:0;width:100%;height:100%;}}</style>
</head><body>{chart_html}<script>
var charts=document.querySelectorAll('div[_echarts_instance_]');
charts.forEach(function(c){{echarts.init(c).resize();}});
window.onresize=function(){{charts.forEach(function(c){{echarts.getInstanceByDom(c).resize();}});}};
<\/script></body></html>"""
```

#### 问题八：with_structured_output 不兼容 DeepSeek

**现象**：`Error code: 400 - 'This response_format type is unavailable now'`

**原因**：`with_structured_output()` 默认使用 `json_schema` 模式（发 `response_format` 参数），DeepSeek API 不支持该参数。

**解决**：改回 `StrOutputParser()`，通过 prompt 约束输出格式 + 正则提取代码块。若需结构化输出，可指定 `method="function_calling"` 使用 tools 参数代替 response_format。

#### 问题九：反思链模型幻觉严重

**现象**：反思链连续 3 轮返回 FAIL，理由包括"numpy 是危险模块"、"set_global_options 是危险函数"等错误判断。

**原因**：反思链同时负责安全检查和代码正确性检查，检查项过多导致模型发散。且模型本身不熟悉 pyecharts 2.x API，无法准确判断代码正确性。

**解决**：去掉 LLM 反思机制，改为：
1. 代码质量靠 `python_chart_chain` 的 prompt 约束（Few-shot 示例）
2. 代码安全靠后续 Docker 沙箱保障
3. 代码执行错误靠 `exec()` 的 try/except 兜底

#### 问题十：StreamingResponse 缺少缓存控制头

**现象**：Nginx 环境下 SSE 流式输出被缓冲，前端无法实时接收数据。

**原因**：`StreamingResponse` 未设置 `Cache-Control` 和 `X-Accel-Buffering` 头。

**解决**：添加 headers，与 AI 助手接口保持一致。

```python
return StreamingResponse(
    generate(),
    media_type="text/event-stream",
    headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"}
)
```

### 后续规划

#### 1. 数据采集机制

建立 `chart_generation_logs` 表，记录每次生成的代码、执行结果、错误信息，为后续模型微调积累训练数据。

```sql
CREATE TABLE chart_generation_logs (
    id INT AUTO_INCREMENT PRIMARY KEY,
    session_id VARCHAR(100),
    user_name VARCHAR(50),
    question TEXT,
    sql_result TEXT,
    generated_code LONGTEXT,
    is_success TINYINT(1),
    error_msg VARCHAR(500),
    model_name VARCHAR(50),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### 2. matplotlib 降级分支

pyecharts 2.x 训练数据较少，模型生成准确率受限。建立 git 分支，切换到 matplotlib 生成静态图表作为对比演示。matplotlib 训练数据充足，生成准确率显著更高，可验证问题出在库的训练数据覆盖度而非系统架构。

#### 3. Docker 安全沙箱

主分支后续集成 Docker，通过进程隔离、文件系统隔离、网络隔离保障代码执行安全。演示分支不启用 Docker 以保证流畅度。
