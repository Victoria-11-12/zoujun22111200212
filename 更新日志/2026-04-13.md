# 2026-04-13 数据分析师模块开发日志

## 一、模块背景

随着系统运行积累了大量用户与 AI 的对话日志，需要对对话质量进行量化评估，为后续模型微调提供数据支撑。今天完成了数据分析师角色的设计与实现，包括角色分流、质量评估系统、数据库权限隔离以及微调数据导出功能。

## 二、角色体系设计

在原有游客、用户、管理员三种角色的基础上，新增数据分析师角色，实现用户分流。数据分析师的定位是分析 LLM 输出质量，为模型微调准备训练数据。

| 角色    | 页面          | 权限                             |
| ----- | ----------- | ------------------------------ |
| 游客    | 数据大屏        | 只能查看大屏展示                       |
| 用户    | 用户端 + AI 问答 | 查询 movies 表，在线绘图               |
| 数据分析师 | 专属分析页面      | 只读 4 张日志表 + 写入评估结果表 + 导出 JSONL |
| 管理员   | 后台管理        | 增删改查 + 回滚                      |

## 三、数据库权限隔离

为数据分析师创建独立的数据库账号，遵循最小权限原则，仅授予必要的权限。即使评估模块被攻击，攻击者也无法访问用户表或修改原始日志数据。

| 表名                      | 权限              | 用途        |
| ----------------------- | --------------- | --------- |
| user\_chat\_logs        | SELECT          | 查询用户对话日志  |
| admin\_chat\_logs       | SELECT          | 查询管理员对话日志 |
| chart\_generation\_logs | SELECT          | 查询绘图日志    |
| security\_warning\_logs | SELECT          | 查询安全警告日志  |
| eval\_results           | SELECT + INSERT | 写入评估结果    |

## 四、质量评估系统

### 4.1 设计思路

采用 LLM-as-Judge 范式，利用大语言模型自身的知识储备对对话记录进行质量评判。该范式的核心思想是：使用能力更强的模型（Judge Model）来评估能力较弱的模型（Target Model）的输出质量。在学术研究中，Zheng et al. (2023) 在论文《Judging LLM-as-a-Judge with ChatGPT》中系统性地验证了 LLM 作为评判者的可行性，研究表明在多数场景下，LLM 的评判结果与人类评判的一致性（Cohen's Kappa）可达 0.8 以上。

本系统中，使用 DeepSeek-R1（推理增强模型）作为评判模型，评估 DeepSeek-V3（对话模型）的输出质量。R1 模型具备更强的逻辑推理和长链思考能力，能够更准确地识别回复中的语义偏差、信息遗漏和格式问题。

评估模块使用独立的 API Key，与用户对话的 API 资源完全隔离，避免评估任务影响用户体验。

### 4.2 两种评估模式

**模式一：回复是否合理（response）**

适用于 user\_chat\_logs、admin\_chat\_logs、security\_warning\_logs 三张表，评估维度包括相关性、完整性、准确性、格式四个方面。

**模式二：代码是否正确（code）**

适用于 chart\_generation\_logs 表，评估维度包括可运行性、图表完整性、工具箱、单位标注、类型匹配五个方面。

### 4.3 评估模型拆分

将初始的统一 EvalResult 模型拆分为两个独立模型，避免维度混淆：

| 模型                 | 用途   | 评估维度                     |
| ------------------ | ---- | ------------------------ |
| ResponseEvalResult | 对话评估 | 相关性、完整性、准确性、格式           |
| CodeEvalResult     | 绘图评估 | 可运行性、图表完整性、工具箱、单位标注、类型匹配 |

### 4.4 评分规则

| 分数     | Verdict | 含义      |
| ------ | ------- | ------- |
| >= 4 分 | pass    | 可用于微调训练 |
| 3 分    | review  | 需要人工复核  |
| <= 2 分 | fail    | 不用于微调   |

### 4.5 评估模型配置

采用"强模型评估弱模型"的设计原则，评估模型与对话模型使用不同的 API Key 和模型实例，实现资源隔离。

| 用途       | 模型                   | 说明                           |
| -------- | -------------------- | ---------------------------- |
| 用户/管理员对话 | deepseek-chat (V3)    | 主 API Key，处理用户请求             |
| 质量评估     | deepseek-reasoner (R1) | 独立 API Key，推理增强模型，用于评分打分 |

### 4.6 异步评估机制

评估任务在后台线程中执行，不阻塞主线程。前端通过轮询 `/api/analyst/evaluate/status` 接口获取实时进度。

## 五、微调数据闭环

### 5.1 完整流程

```
第一步：用户使用系统产生对话日志
  → 记录到 user_chat_logs、admin_chat_logs、chart_generation_logs、security_warning_logs

第二步：数据分析师触发质量评估
  → DeepSeek-R1 对每条对话记录进行多维度评分，结果存入 eval_results

第三步：筛选高质量数据并导出
  → 筛选 score >= 4 且 verdict = pass 的记录，导出 JSONL

第四步：上传至微调平台训练
  → 阿里云百炼或 DeepSeek 开源工具链执行监督微调（SFT）

第五步：替换模型 ID
  → 系统配置中替换 MODEL_NAME，重启服务
```

### 5.2 JSONL 导出格式

导出的 JSONL 文件符合 DeepSeek 微调平台的标准格式，每行一条训练数据，包含 user 和 assistant 两个角色的对话内容。导出时自动过滤安全拦截类对话，只保留正常业务对话和绘图代码。

```jsonl
{"messages": [{"role": "user", "content": "查询票房前十"}, {"role": "assistant", "content": "票房前十如下：..."}]}
```

### 5.3 筛选条件

- 最低评分筛选（默认 4 分）
- 数据来源多选（用户对话、管理员对话、安全警告、图表生成）
- 时间范围筛选
- 自动过滤安全拦截类对话（排除包含"警告""封禁""安全威胁"等关键词的记录）

## 九、遇到的关键问题与解决方案

### 问题一：with\_structured\_output 不兼容 DeepSeek

**现象**：使用 with\_structured\_output 时报错"This response\_format type is unavailable now"。

**原因**：LangChain 的 with\_structured\_output 默认使用 json\_schema 模式（通过 response\_format 参数），DeepSeek API 暂不支持该参数。尝试 function\_calling 模式时，deepseek-reasoner 也不支持 tool\_choice 参数。

**解决**：改用 json\_mode 方法，通过 `with_structured_output(method="json_mode")` 指定 LangChain 使用 response\_format 通道而非 tool\_calling 通道。同时安装 langchain-deepseek 依赖，使用 ChatDeepSeek 替代 ChatOpenAI，确保与 DeepSeek API 的完全兼容。

### 问题二：deepseek-reasoner 输出格式化分析文本

**现象**：deepseek-reasoner 在评估时输出带有表格、加粗、标题的 Markdown 格式分析文本，而非纯 JSON。

**原因**：deepseek-reasoner 内置了 Chain-of-Thought（CoT）推理机制，模型在输出前会进行长链思考，倾向于生成结构化的分析报告。这是推理增强模型的固有特性，无法通过 prompt 约束完全消除。

**解决**：在 prompt 中明确要求输出纯 JSON 格式，并在 prompt 末尾提供 JSON 输出示例。同时在代码层面增加正则提取逻辑，从模型输出中提取 JSON 内容块，确保解析的鲁棒性。

### 问题三：评估维度混乱

**现象**：对话评估结果中出现了绘图评估的维度（toolbox、unit\_label 等），且 score 与 verdict 不匹配。

**原因**：初始设计使用统一的 EvalResult 模型，所有维度均为 Optional，模型自由发挥填写了不属于当前评估类型的维度。

**解决**：拆分为 ResponseEvalResult 和 CodeEvalResult 两个独立模型，在 prompt 中明确列出各自的维度名称和计算规则，消除模型自由发挥的空间。

### 问题四：导出的 JSONL 包含安全拦截对话

**现象**：导出的数据中包含大量安全拦截类对话（SQL 注入、社会工程攻击等）。

**原因**：安全拦截的对话评分较高（AI 回复专业且格式规范），被筛选为"高质量"数据。但这类对话不适合用于微调，否则会导致模型在正常场景下也倾向于拒绝回复。

**解决**：在导出时过滤 ai\_content 中包含"警告""封禁""安全威胁"等关键词的记录，只保留正常业务对话。

### 问题五：对话量趋势图日期错位

**现象**：12 日数据显示 11 日数据，11 日显示 10 日数据。

**原因**：MySQL `DATE()` 函数处理 `DATETIME` 时存在时区转换问题。

**解决**：改用 `DATE_FORMAT(created_at, '%Y-%m-%d')` 替代 `DATE(created_at)`。

### 问题六：图表初始化错误

**现象**：ECharts 提示图表实例已初始化 / 容器宽高为 0。

**解决**：初始化前调用 `dispose()` 销毁旧实例，将图表初始化延迟到数据加载完成后执行。

## 十二、后续规划

- **并发评估**：通过 asyncio.Semaphore 实现多条并发评估，提升速度
- **消息队列**：引入 RabbitMQ 实现异步解耦，评估任务入队按优先级调度
- **MCP 升级**：当业务量增大、数据源增多时，可基于 MCP 协议重构为交互式评估服务
- **微调执行**：将导出的 JSONL 上传至阿里云百炼或 DeepSeek 开源工具链执行监督微调
- **效果对比**：对比微调前后同一批问题的回复质量评分，量化微调效果
