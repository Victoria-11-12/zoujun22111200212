## 二、票房预测模块重构

### 2.1 LightGBM 模型集成

在原有随机森林模型的基础上，新增 LightGBM 模型用于深度票房预测。两个模型各司其职：

| 功能 | 模型 | 特征数量 | 用途 |
|------|------|---------|------|
| 黑马票房榜 | Random Forest | 9 个 | 识别低预算高回报电影 |
| 深度票房预测 | LightGBM | 4 个 | 用户交互式预测 |

模型加载采用独立 try-except 结构，确保单个模型加载失败不影响另一个：

```python
try:
    rf_model = joblib.load('random_forest_model.pkl')
    print("随机森林模型加载成功！")
except Exception as e:
    print(f"随机森林模型加载失败: {e}")
    rf_model = None

try:
    lgb_model = joblib.load('lightgbm_model_1.pkl')
    print("LightGBM模型加载成功！")
except Exception as e:
    print(f"LightGBM模型加载失败: {e}")
    lgb_model = None
```

### 2.2 预测表单简化

将深度票房预测表单从 9 个输入框简化为 4 个核心参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| budget | 数字输入 | 电影预算（默认值 0） |
| genres | 下拉选择 | 电影类型（21 种） |
| New_Director | 下拉选择 | 是否新导演（Yes/No） |
| New_Actor | 下拉选择 | 是否新演员（Yes/No） |

深度预测接口 `POST /api/flask/predict_deep`：

```python
@app.route('/api/flask/predict_deep', methods=['POST'])
def predict_deep():
    if lgb_model is None:
        return jsonify({"code": 500, "msg": "LightGBM模型未加载"})

    data = request.json
    input_df = pd.DataFrame({
        'genres': [data.get('genres', '')],
        'New_Director': [data.get('New_Director', 'No')],
        'New_Actor': [data.get('New_Actor', 'No')],
        'budget': [float(data.get('budget', 0))]
    })

    # 类别特征编码
    for col in ['genres', 'New_Director', 'New_Actor']:
        input_df[col] = pd.factorize(input_df[col])[0]

    predicted = lgb_model.predict(input_df)[0]
    return jsonify({"code": 200, "predicted_gross": float(predicted)})
```

### 2.3 ROI 对比可视化

新增 `GET /api/flask/roi_comparison` 接口，提供真实 ROI 与预测 ROI 的散点图对比：

- **X 轴**：真实 ROI（gross / budget）
- **Y 轴**：预测 ROI（随机森林预测票房 / budget）
- **对角线**：y = x 完美预测线
- **交互**：鼠标悬停显示电影名称和具体 ROI 值

数据过滤策略：

| 过滤条件 | 说明 |
|---------|------|
| budget > 0 且 gross > 0 | 排除无效数据 |
| actual_roi <= 100 | 排除极端异常值 |
| predicted_roi <= 100 | 排除预测异常值 |

过滤后保留 4412 条有效数据（原始 7598 条 → 过滤预算票房后 4458 条 → 移除 46 条极端异常值）。

## 三、黑马筛选模块优化

### 3.1 模型切换

将黑马筛选功能从 LightGBM 切换到随机森林模型。原因：随机森林使用 9 个特征（含 Facebook 点赞数、投票数等社交指标），对低预算电影的预测能力更强，更适合识别"黑马"。

### 3.2 特征准备函数重构

```python
def prepare_features(df):
    df = df.copy()
    feature_columns = [
        'budget', 'director_facebook_likes', 'actor_1_facebook_likes',
        'actor_2_facebook_likes', 'actor_2_facebook_likes', 'movie_facebook_likes',
        'num_voted_users', 'num_user_for_reviews', 'imdb_score'
    ]
    for col in feature_columns:
        if col in df.columns:
            df[col] = pd.to_numeric(df[col], errors='coerce').fillna(0)
        else:
            df[col] = 0
    return df[feature_columns]
```

### 3.3 对数转换处理

随机森林模型训练时对目标变量进行了对数转换（`np.log1p`），预测时需反向转换：

```python
predictions_log = rf_model.predict(df_features)  # 模型输出对数值
predictions = np.expm1(predictions_log)           # 转换回原始票房
```

### 3.4 修复的问题

**特征名称不匹配**：训练代码中 `actor_2_facebook_likes` 重复出现（应为 `actor_3_facebook_likes`），导致推理时报错 "Feature names unseen at fit time"。将推理代码的特征列表与训练代码保持一致，确保匹配。

## 四、SQL Agent 查询效率优化

### 4.1 优化背景

SQL Agent 是用户查询电影数据的核心组件，基于 LangChain 的 ReAct Agent 框架实现。优化前，一次简单查询（如"帮我查询战狼这部电影的数据"）耗时 30-40 秒，用户体验较差。

### 4.2 性能瓶颈分析

以"帮我查询战狼这部电影的数据"为例，优化前的完整执行链路：

```
1. sql_db_list_tables       → LLM 思考 3-5s → 返回 ["movies"]
2. sql_db_schema            → LLM 思考 3-5s → 返回 movies 表结构
3. sql_db_query_checker     → LLM 思考 3-5s → 判断 SQL 安全性
4. sql_db_query             → LLM 思考 3-5s → 执行第一次查询
5. sql_db_query_checker     → LLM 思考 3-5s → 判断第二次 SQL 安全性
6. sql_db_query             → LLM 思考 3-5s → 执行第二次查询
7. 生成最终回复              → LLM 思考 3-5s → 输出格式化结果
```

时间分布：

| 环节 | 耗时 | 占比 |
|------|------|------|
| LLM 调用（6 次） | 24-30 秒 | 80-90% |
| 数据库查询（2 次） | < 0.2 秒 | < 1% |
| 网络传输 | 1-2 秒 | 5-10% |

**结论：瓶颈不在数据库查询，而在 LLM 调用次数过多。**

### 4.3 冗余工具分析

原始方案使用 `SQLDatabaseToolkit` 自动注册 4 个工具，其中 3 个是冗余的：

| 工具 | 作用 | 是否冗余 | 原因 |
|------|------|---------|------|
| sql_db_list_tables | 查询数据库有哪些表 | 冗余 | 系统只有一张 movies 表，每次返回结果相同 |
| sql_db_schema | 查询表结构 | 冗余 | 表结构固定，可预先注入 prompt |
| sql_db_query_checker | 检查 SQL 安全性 | 冗余 | 系统已有独立的安全防御体系 |
| sql_db_query | 执行 SQL | 必要 | 唯一执行实际查询的工具 |

### 4.4 优化方案

**方案一：表结构预注入**

将 movies 表的完整 DDL 直接写入 Agent 的 system prompt，省去 list_tables 和 schema 两个工具调用。

**方案二：精简工具集**

使用 `@tool` 装饰器手动创建唯一的 `sql_db_query` 工具，不再使用 `SQLDatabaseToolkit`，从物理层面移除冗余工具。

**方案三：限制最大推理轮次**

将 `max_iterations` 从 4 降为 2，强制 Agent 在 2 轮内完成任务。

**方案四：查询规则约束**

在 prompt 中明确查询规则（禁止 SELECT *、强制 WHERE、强制 LIMIT 等），防止低效操作。

### 4.5 代码实现

```python
from langchain.tools import tool

@tool
def sql_db_query(query: str) -> str:
    """执行 SQL 查询语句，输入完整的 SQL 语句"""
    return db_user.run(query)

user_toolkit = [sql_db_query]

SQL_PROMPT = ChatPromptTemplate.from_messages([
    ("system", """你是一个专业的 SQL 查询助手。数据库中有一张 movies 表，结构如下：
CREATE TABLE movies (
    id INT PRIMARY KEY AUTO_INCREMENT,
    movie_title VARCHAR(255),
    director_name VARCHAR(255),
    actor_1_name VARCHAR(255),
    actor_2_name VARCHAR(255),
    actor_3_name VARCHAR(255),
    genres VARCHAR(255),
    title_year INT,
    imdb_score DECIMAL(3,1),
    gross BIGINT,
    budget BIGINT,
    duration INT,
    language VARCHAR(100),
    country VARCHAR(100),
    content_rating VARCHAR(255)
);
查询规则：
1. 只使用 SELECT 语句，禁止 INSERT、UPDATE、DELETE、DROP、TRUNCATE、ALTER
2. 禁止 SELECT *，必须明确列出需要的字段名
3. 必须包含 WHERE 条件，禁止全表扫描
4. 查询结果必须包含 LIMIT，默认 LIMIT 20，不超过 LIMIT 100
5. 查询中文电影时使用 LIKE '%关键词%'
6. 直接生成 SQL 并执行，不需要先查表结构或验证 SQL
7. 只调用一次 sql_db_query，不要重复查询"""),
    ("human", "{input}"),
    MessagesPlaceholder("agent_scratchpad")
])

sql_agent = create_tool_calling_agent(llm=llm, tools=user_toolkit, prompt=SQL_PROMPT)
sql_executor = AgentExecutor(
    agent=sql_agent,
    tools=user_toolkit,
    verbose=True,
    max_iterations=2,
    handle_parsing_errors=True
)
```

### 4.6 优化效果

**执行流程对比：**

```
优化前（6 次 LLM 调用）：
list_tables → schema → query_checker → query → query_checker → query → 生成回复

优化后（1-2 次 LLM 调用）：
query → 生成回复
```

**性能对比：**

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| LLM 调用次数 | 6 次 | 1-2 次 | 减少 67-83% |
| 响应时间 | 30-40 秒 | 10-15 秒 | 提升 2-3 倍 |
| 工具数量 | 4 个 | 1 个 | 精简 75% |
| 最大推理轮次 | 4 轮 | 2 轮 | 减少 50% |

**安全性保障：** 即使去掉 query_checker，系统安全性不受影响。数据库层面使用只读账号（readonly_user 只有 SELECT 权限），且七层安全防御体系在 Agent 之前已拦截危险输入。

### 4.7 技术原理

**ReAct 范式：** SQL Agent 基于 ReAct（Reasoning + Acting）范式，核心是一个 while 循环：Thought（思考）→ Action（调用工具）→ Observation（观察结果），直到输出 Final Answer。

**Tool Calling 机制：** 使用 `create_tool_calling_agent` 创建的 Agent 通过模型原生的 Function Calling 能力调用工具，相比传统文本模式 ReAct Agent，减少了文本解析开销，提升了调用效率和准确性。

**优化核心思想：** 减少 Agent 循环中的 LLM 调用次数。将静态信息预注入 prompt、移除冗余工具、限制推理轮次，从 6 次 LLM 调用压缩到 1-2 次。

## 五、前端优化

### 5.1 表单样式优化

- 更大的输入框内边距（12px 16px）

- 圆角设计（6px）
- 悬停和聚焦交互效果

### 5.2 性能修复

使用展开运算符 `...` 处理 4000+ 数据导致栈溢出，改用 `forEach` 循环：

```javascript
// 错误写法（栈溢出）
const maxROI = Math.max(...res.data.map(item => Math.max(item.actual_roi, item.predicted_roi)));

// 正确写法
let maxROI = 0;
res.data.forEach(item => {
    const maxVal = Math.max(item.actual_roi, item.predicted_roi);
    if (maxVal > maxROI) maxROI = maxVal;
});
```
