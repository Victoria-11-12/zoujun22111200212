# 2026-04-18 FastAPI 服务架构升级日志

## 一、升级背景

随着系统功能的不断迭代，FastAPI 服务（app3.py）暴露出几个架构层面的问题。首先，绘图模块（CodeAct Agent）采用线性流程，LLM 生成的 pyecharts 代码经常出现格式问题，一次成功率不高，用户遇到失败后需要重新发起请求，体验较差。其次，评估模块的并发能力有限，信号量为 3，面对大量日志记录时评估速度偏慢。第三，SQL Agent 仍使用 LangChain 内置的 SQLDatabaseToolkit，自动注册了 4 个冗余工具，导致每次查询需要 6 次 LLM 调用，响应时间长达 30-40 秒。

针对以上问题，本次更新对 FastAPI 服务进行了四项重大改造：绘图模块引入 LangGraph 状态机实现自动重试、评估模块提升并发能力并优化代码结构、SQL Agent 精简工具集、管理员 Agent 合并冗余工具。

## 二、LangGraph 状态机设计

### 2.1 设计思路

旧版绘图模块的执行流程是线性的：意图判断 → SQL Agent 查数据 → LLM 生成 pyecharts 代码 → 安全检查 → Docker 沙箱执行。这个流程中，后三步（代码生成、安全检查、沙箱执行）任何一步失败都会直接返回错误，没有重试机会。实际使用中，LLM 生成的代码经常缺少 `CHART_HTML` 赋值、使用了禁止的 `import os` 等，一次通过率大约只有 60%-70%。

为了解决这个问题，引入了 LangGraph 的 `StateGraph`，将线性流程重构为带条件路由的状态机。核心思想是：在代码生成和沙箱执行之间插入一个静态评估节点（eval），如果评估不通过，将具体问题作为 feedback 传回代码生成节点，让 LLM 根据反馈修改代码后重新生成，最多重试 3 次。

### 2.2 状态定义

使用 TypedDict 定义全局状态，各节点通过读写同一个状态字典进行通信：

```python
class ChartGraphState(TypedDict, total=False):
    question: str       # 用户输入
    session_id: str     # 会话 ID
    user_name: str      # 用户名
    sql_result: str     # SQL Agent 输出
    code_raw: str       # Python Agent 生成的原始代码
    code: str           # 沙箱内执行的纯净 Python 代码
    feedback: str       # 反馈信息（用于重试时指导代码修改）
    eval_pass: bool     # 静态评估是否通过
    attempts: int       # 尝试计数器（防止无限循环）
    chart_html: str     # 最终渲染的 HTML
    error: str          # 错误信息
```

其中 `feedback` 和 `attempts` 是重试机制的关键字段：feedback 携带上一次失败的具体原因，attempts 记录已尝试次数，达到 3 次上限后强制终止。

### 2.3 节点设计

状态机包含四个节点，各司其职：

| 节点 | 函数 | 功能 |
| ---- | ---- | ---- |
| sqlagent | `_node_sqlagent` | 调用 SQL Agent 查询 movies 表，结果写入 `sql_result` |
| pythonagent | `_node_pythonagent` | 根据 `sql_result` 和 `feedback` 生成 pyecharts 代码 |
| eval | `_node_eval` | 静态检查代码安全性和格式约定，输出 `eval_pass` 和 `feedback` |
| pyecharts_sandbox | `_node_pyecharts_sandbox` | Docker 沙箱执行代码，提取 HTML 写入 `chart_html` |

### 2.4 条件路由

两个路由函数控制流程走向。eval 节点之后的路由决定代码是否需要重新生成，pyecharts_sandbox 节点之后的路由决定沙箱执行失败后是否重试：

```python
def _route_after_eval(state):
    if not state.get("eval_pass", False):
        if int(state.get("attempts", 0)) >= 3:
            state["error"] = "代码检查失败且达到最大重试次数"
            return END
        return "pythonagent"       # 携带 feedback 重试
    return "pyecharts_sandbox"     # 通过，进入沙箱

def _route_after_sandbox(state):
    if state.get("chart_html"):
        return END                 # 成功
    if state.get("error"):
        if int(state.get("attempts", 0)) >= 3:
            return END             # 超过重试次数
        return "pythonagent"       # 沙箱失败，重试
    state["error"] = "未知的图表状态"
    return END
```

### 2.5 工作流构建

```python
def _build_chart_graph():
    graph = StateGraph(ChartGraphState)
    graph.add_node("sqlagent", _node_sqlagent)
    graph.add_node("pythonagent", _node_pythonagent)
    graph.add_node("eval", _node_eval)
    graph.add_node("pyecharts_sandbox", _node_pyecharts_sandbox)

    graph.set_entry_point("sqlagent")
    graph.add_edge("sqlagent", "pythonagent")
    graph.add_edge("pythonagent", "eval")
    graph.add_conditional_edges("eval", _route_after_eval)
    graph.add_conditional_edges("pyecharts_sandbox", _route_after_sandbox)

    return graph.compile()

chart_graph = _build_chart_graph()
```

sqlagent → pythonagent → eval 这两段是固定边，始终按顺序执行。eval 和 pyecharts_sandbox 之后是条件边，根据状态决定下一步走向 pythonagent（重试）还是 END（终止）。

### 2.6 完整流程

```
sqlagent → pythonagent → eval ——(不通过)——→ pythonagent（重试，最多3次）
                              ——(通过)——→ pyecharts_sandbox ——(失败)——→ pythonagent（重试）
                                                        ——(成功)——→ END
```

### 2.7 静态评估增强

新版 `_static_eval` 函数相比旧版检查更加严格，同时返回结构化的反馈信息用于指导重试：

| 检查项 | 旧版 | 新版 |
| ---- | ---- | ---- |
| CHART_HTML 赋值 | 检查 | 检查 |
| CHART_HTML_START/END 标记 | 检查 | 检查 |
| 允许的 import | pyecharts, pandas, numpy, json | 仅 pyecharts |
| 禁止的令牌 | 无 | os, sys, subprocess, socket, requests, urllib, open, eval, exec, \_\_import\_\_ |
| 返回值 | 布尔值 | (passed, issues, feedback) 结构化元组 |

## 三、评估模块优化

### 3.1 并发架构设计

评估模块采用**后台线程 + 异步协程**的混合并发模型。评估任务是一个长时间运行的后台任务（可能需要几分钟），不能阻塞 FastAPI 的事件循环，因此通过 `threading.Thread` 将评估任务放到后台线程中执行，HTTP 请求立即返回。在线程内部，使用 `asyncio.run()` 创建新的事件循环，通过 `asyncio.gather()` 并发执行多个评估任务，充分利用 I/O 等待时间。

```
HTTP 请求（FastAPI 异步路由）
    │
    ├── start_evaluation 路由
    │       ├── 初始化 eval_progress（线程锁保护）
    │       └── threading.Thread(target=run_evaluation_task)
    │               └── asyncio.run(evaluate_records_task_async())
    │                       ├── asyncio.Semaphore(5)
    │                       └── asyncio.gather(*[eval_one(r) for r in records])
    │                               └── eval_llm.ainvoke()
    │
    └── get_evaluate_status 路由（前端轮询）
            └── 读取 eval_progress（线程锁保护）
```

### 3.2 线程锁设计

```python
eval_lock = threading.Lock()
eval_progress = {"status": "idle", "total": 0, "completed": 0}
```

`eval_lock` 保护 `eval_progress` 全局字典的读写。前端通过轮询 `/api/analyst/evaluate/status` 接口获取实时进度，后台线程通过更新 `eval_progress` 反映评估进展，两者存在并发读写，需要线程锁保证数据一致性。

| 场景 | 线程锁作用 |
| ---- | --------- |
| start_evaluation 初始化进度 | 加锁写入，防止与轮询读取冲突 |
| evaluate_records_task_async 更新完成数 | 加锁更新 completed，保证原子性 |
| get_evaluate_status 读取进度 | 加锁读取，防止读到中间状态 |

### 3.3 并发能力提升

| 指标 | 旧版 | 新版 | 说明 |
| ---- | ---- | ---- | ---- |
| 信号量 | Semaphore(3) | Semaphore(5) | 并发评估数从 3 提升到 5 |
| eval_one 定义 | 嵌套函数 | 顶层函数 | 提取为独立函数，结构更清晰 |
| 进度初始化 | 路由和函数内重复设置 | 仅在路由中初始化 | 消除重复逻辑 |

### 3.4 评估模型切换

## 四、SQL Agent 精简

从 LangChain 内置的 `create_sql_agent`（自动注册 4 个工具）切换为手动创建的单工具 Agent。将 movies 表的完整 DDL 直接写入 system prompt，省去 list_tables 和 schema 两个工具调用，从物理层面移除冗余工具。

```python
# 旧版：4 个工具（list_tables, schema, query_checker, query）
user_toolkit = SQLDatabaseToolkit(db=db_user, llm=llm)
sql_agent = create_sql_agent(llm=llm, toolkit=user_toolkit, ...)

# 新版：1 个工具
@tool
def sql_db_query(query: str) -> str:
    """执行 SQL 查询语句，输入完整的 SQL 语句"""
    return db_user.run(query)

sql_agent = create_tool_calling_agent(llm=llm, tools=[sql_db_query], prompt=SQL_PROMPT)
sql_executor = AgentExecutor(agent=sql_agent, tools=[sql_db_query], ...)
```

| 指标 | 旧版 | 新版 |
| ---- | ---- | ---- |
| 工具数量 | 4 个 | 1 个 |
| LLM 调用次数 | 6 次 | 1-2 次 |
| 响应时间 | 30-40 秒 | 10-15 秒 |

## 五、管理员 Agent 优化

### 5.1 工具精简

移除了 `rollback_last` 工具，统一使用 `rollback_batch` 进行回滚操作。工具数量从 5 个减少到 4 个：create_user、safe_execute_sql、rollback_batch、start_batch。同时将 `max_iterations` 从默认值 15 显式设为 10，避免管理员复杂指令导致推理轮次过多。

### 5.2 备份函数合并

旧版有两个备份函数（`backup_before_modify` 处理 DELETE/UPDATE、`backup_insert` 处理 INSERT），新版合并为一个 `backup_data` 函数，统一处理三种操作的备份逻辑，代码更加简洁。
